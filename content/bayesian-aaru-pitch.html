Turning scenario generators into calibrated prediction machines

The Architecture Gap

Current Model

Run simulation → get prediction → decision → observe outcome → run new simulation

Bayesian Workflow

Agents carry posterior distributions that update with each observation. The model learns, not just reruns.

Why This Matters

1. Agents Become Learners
Currently: synthetic agents have fixed preference parameters (or sampled from a distribution).

With persistent Bayesian updating: each agent's preferences have uncertainty that shrinks as you observe behavior.

First-time Horizon shopper: wide priors on lobster preference. After 3 visits with no seafood purchases: tight posterior → probably not your lobster customer.
2. The Simulation Becomes a Living Model
Instead of "run simulation, get answer, done" → the simulation accumulates knowledge.

Each Horizon transaction updates the population's belief state. You're not starting from scratch each time.
3. Partial Pooling = Competitive Moat
New store opening? You don't have data on those customers, but you have posteriors from similar demographics.

Hierarchical Bayesian structure: individual agents inherit from group-level distributions.

This is exactly how PRISM archetypes work → Aaru is already halfway there architecturally.

The Lobster Example

Static Simulation
"Given our agent parameters, 23% of simulated customers would buy lobster. Stock it."
Bayesian Version
Initial forecast: 23% expected purchase rate, but 90% CI: [14%, 32%]
After week 1: Observed 18% → posterior tightens to [15%, 24%]
Diagnosis: High-uncertainty agents (new to neighborhood) drove the miss — their posteriors are updating fast
Week 4 forecast: 21% ± 3% — much tighter because the model learned
